# PPT Composer 用户指南

这份文档面向第一次使用 PPT Composer 的用户。你不需要先理解所有 CLI 和内部文件，只要记住：先确认规则，再生成图片，再复审返工，最后组装 PPTX。

![PPT Composer 使用流程](images/workflow-overview.png)

## 一句话理解

PPT Composer 不是直接把一堆文字塞进 PowerPoint。它先把你的需求、参考资料、每页内容和风格规则写成一个可审阅的 `deck-protocol.json`，等你确认后，再为每一页生成一张完整 PNG，最后把这些 PNG 组装成 PPTX。

最终 PPTX 的每页只有一张图。标题、正文、图表、标签、logo 都在这张图里，不会再在 PowerPoint 里后期叠字。

## 你需要记住的 4 个东西

| 名称 | 你怎么理解 |
| --- | --- |
| `deck-protocol.json` | 生图前的规则书。它写清楚页数、风格、每页标题、claim、参考资料、保真模式和最终 prompt。 |
| `imagegen-jobs.json` | 每页图片生成状态。它只记录执行过程，不替代 `deck-protocol.json`。 |
| `visual-qa.json` | 组装前检查报告。包括 PNG 是否存在、是否太小、是否 placeholder，以及视觉复审结果。 |
| `png-manifest.json` | 最终组装清单。只有每页都有可接受 PNG 后才会创建。 |

如果你只想正常使用，最重要的是看懂 `deck-protocol.json` 的摘要，并在生图前确认它。

## 为什么要先确认协议

生图比改文字贵，也更慢。先确认协议可以避免这些问题：

- 页数、标题、核心观点还没对齐就开始生成。
- 参考图、logo、表格没有绑定到正确页面。
- 本来要严格保留的数字、图注、曲线被模型自由发挥。
- 生成后才发现风格、受众、语言或输出路径不对。

所以正常流程是：

1. 你提供需求和参考资料。
2. Codex 创建 `deck-protocol.json`。
3. 你看协议摘要，提出修改。
4. Codex 用协议补丁工具更新协议。
5. 你明确确认协议。
6. Codex 才开始按页生成 PNG。

## 三种保真模式

| 模式 | 适合场景 | 规则 |
| --- | --- | --- |
| `free` | 没有强参考，只按主题和风格生成。 | 可以自由设计，但不能偏离已确认主题。 |
| `light_redraw` | 有参考内容，但允许重排和美化。 | 事实、趋势、关键数字要保留。 |
| `strict_embed` | 论文图、表格、logo、关键截图等必须忠实保留。 | 数字、曲线、表头、logo、图注不能乱改。 |

如果你不确定，用自然语言告诉 Codex：哪些页面必须严格保留资料，哪些页面可以自由设计。

## 视觉复审和按页返工

生成图片后，PPT Composer 会先做确定性检查，再做视觉复审。视觉复审主要看三件事：是否和整套 deck 一致、是否偏离协议、是否有明显生成图问题。

![视觉复审与按页返工](images/visual-review-revision-loop.png)

视觉复审的职责划分是：

| 角色 | 做什么 |
| --- | --- |
| 主代理 | 负责确定性 QA、读协议、记录状态、决定是否返工、组装 PPTX。 |
| `vision` / reviewer subagent | 多页 deck 中优先负责看图，对照协议给出 `pass`、`warn`、`fail`。 |

复审结果含义：

| 结果 | 含义 |
| --- | --- |
| `pass` | 可以进入最终组装。 |
| `warn` | 可用，但有小问题，会记录警告。 |
| `fail` | 需要返工，不能组装。 |

返工默认只处理问题页，不会重做整份 PPT。正确返工顺序是：

1. 找到失败页和失败原因。
2. 必要时先修改 `deck-protocol.json` 的 prompt、负面约束、layout intent 或保真绑定。
3. 标记这一页为 `revision_requested`。
4. 只重新生成这一页 PNG。
5. 新 PNG 回填为新的 attempt。
6. 再次视觉复审。

## 常用对话方式

### 从参考资料生成 PPT

```text
使用 ppt-composer:image-first-ppt。
基于 reference/paper.pdf 和 reference/logo.png 生成 10 页中文科研汇报 PPT。
风格：高端科研咨询，16:9，中文主语言，关键标题可中英双语。
第 4-6 页要严格保留论文图和关键数字。
输出到 output/report.pptx。
```

### 修改协议

```text
第 3 页标题改成“核心实验结果”，claim 聚焦 sample efficiency。
```

```text
第 6 页改成 strict_embed，并绑定 fig-3。
```

```text
第 8 页不要使用复杂 3D 图，改成简洁流程图风格。
```

### 确认协议后开始生成

```text
协议确认，可以开始生成图片并组装 PPTX。
```

### 发现某页不好

```text
第 5 页风格和其他页面不一致，标题也偏离协议。只返工第 5 页，保持原来的 claim 和参考图不变。
```

```text
第 7 页表格数字看不清，先增强 prompt，要求表格标题和关键数字清晰，然后只重生第 7 页。
```

## 常见误解

| 误解 | 实际规则 |
| --- | --- |
| “PPT 里还能随便编辑每个文本框。” | 当前 image-first 输出是一页一张 PNG，低可编辑性，但展示效果稳定。 |
| “可以先生成背景，再在 PPT 里叠字。” | 不可以。标题、正文和标签必须直接渲染在 PNG 里。 |
| “图片不好就整份 PPT 重来。” | 不需要。默认按页返工，只重生问题页。 |
| “`imagegen-jobs.json` 可以当内容计划改。” | 不可以。内容真相在 `deck-protocol.json`。 |
| “视觉 QA 只看文件存在。” | 现在分两层：确定性 PNG 检查 + 视觉复审。视觉复审看一致性、协议偏离和基础图像问题。 |

## 什么时候应该停下来检查

在这些节点应该先看一眼结果：

- `deck-protocol.json` 生成后：检查页数、标题、claim、引用资料、保真模式。
- 图片生成后：检查每页是否风格一致、文字清楚、没有跑题。
- 视觉复审失败后：先看失败原因，再决定是改 prompt、改参考绑定，还是只重生。
- PPTX 组装后：确认最终 QA 显示一页一图、没有 PowerPoint 文本叠加。

## 你可以不管的内部细节

大多数时候你不需要手动运行 CLI。你只要用自然语言告诉 Codex：

- 要生成什么 deck。
- 用哪些参考资料。
- 哪些页面要严格保留资料。
- 哪些页面需要修改。
- 是否确认协议。
- 哪些页需要返工。

Codex 会在内部调用协议补丁、视觉复审、返工和组装工具。
